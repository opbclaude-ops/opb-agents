## 1. Контекст и предыстория (читать целиком, не сокращать)

### 1.1 Кто пользователь

- **Сметчик ОПБ** (отдел подготовки бюджетов) компании **Авентин**.
- Считает наряды на монтаж слаботочки **после выигрыша тендера** —
  производственный расчёт, для оплаты бригадам.
- Аккаунт Drive: `opbclaude@gmail.com` (отдельный, под работу с Claude).
- С телефона Samsung S20 FE — короткие сообщения, иногда с опечатками.

### 1.2 Слаботочные системы в скоупе

АППЗ, СВН, ТСО, СКУД, СОУЭ, ДП, ЭОМ. Объекты — ЖК и паркинги.

### 1.3 Терминология (обязательно)

- **Единая расценка** — одна цена работы, общая для ОПБ и КД. Лист
  «Справочник» в корпоративном файле.
- **ВСП** — вспомогательные работы (штроба, отверстия, узлы, ПНР).
  Не входят в наряды напрямую, считаются по методике.
- **ПНР** — пусконаладочные работы. Часть ВСП-семейства, считаются по
  методике из листа «Методика» файла `База_3_ОПБ`.
- **Тождество корректности**: единая расценка по спеке + ВСП по методике
  = сумма фактических нарядов из **Синтеки**. Для АППЗ точность 1,02–1,05.
- **Синтека** — учётная ERP Авентина, источник истины, **есть API**.
- **Пакет** — наряд с сопутствующими работами в одной строке (штроба +
  гофра + кабель + монтаж). В корпоративном файле раскладывается формулой.
- **Классификатор оборудования** — компактный реестр «марка → группа →
  наряд». Растёт из практики.

### 1.4 Объекты (всё в `Проекты компании/`)

- **ПП5** (Пульс 5) — пилот, есть полный комплект документации.
- **Сэтл Лофт** — на нём прогонялся аудит ПНР, есть готовая «Спец Лофт»
  с фактическими цифрами (точность 104,8%).
- **КК33** (Красный Кирпичник 33, 35), **Ривьера**, **Астра 4** — другие
  объекты, разной степени проработки.

### 1.5 Жёсткие правила и стоп-зоны (нарушать нельзя)

- **КД (Коммерческий Департамент) — стоп-зона.** Файл «Шаблон расчета КД
  с ПНР.xlsx» — для понимания, **не упоминать в навыках/агентах**, не
  использовать как источник ставок.
- **`userPreferences`:** не отслеживать местоположение, не передавать
  геоданные. Жёсткое правило.
- **Не выдумывать риски** про корпоративные системы пользователя (Синтека,
  корп.файл, политика) — он знает свою инфраструктуру лучше.
- **Не оценивать инженерные решения свысока.**
- **Один уточняющий вопрос максимум.** Лучше ноль. Пользователь ведёт сам.
- **Длинные плотные ответы — ок, если по делу. Тон «ты».**
- **Сначала артефакт, потом обсуждение.** Любит когда дают скачать файл,
  потом разговор.
- **Не дублировать самопроверки навыков, не «улучшать» их логику.**
  Навык — источник истины, агент — транспорт/координатор.

**Pipeline типового объекта (целевой):**

1. **Сбор данных** (параллельно где можно):
   - `agent-shahmatka` → шахматка АППЗ (этажи × секции, типы помещений).
   - `agent-kvartirograph` → типы квартир (студии/1к/2к/3к).
   - `agent-spec` → спецификация (по одной или сразу всем 7 системам).
2. **Расчёт**:
   - `agent-cable` → раскладка кабеля, наряды на монтаж кабеля.
   - `agent-equip` → наряды на установку оборудования (приборы, шкафы).
   - `agent-vsp-pnr` → ВСП (штроба, отверстия, узлы) + ПНР (22 строки
     методики для АППЗ).
3. **Сводка** (внутри Начальника):
   - Складывание единой расценки + ВСП.
   - Сверка с фактом из Синтеки (если есть выгрузка).
   - Отчёт пользователю с артефактами.

**Связи между агентами слабые.** Каждый автономен, может работать без
предшественников, но при их наличии — использует результат как ускорение.
**Прямых жёстких зависимостей нет.**

## 3. Что должен делать Начальник

### 3.1 Роль и границы

**Роль:** оркестратор/диспетчер. Управляет жизненным циклом расчёта по
объекту: понимает запрос, выбирает сабагентов, последовательно/параллельно
запускает, собирает результаты, сводит, отчитывается.

**НЕ делает сам:**

- Не считает наряды (это работа расчётчиков).
- Не парсит PDF (это работа `agent-spec`).
- Не определяет типы квартир (это `agent-kvartirograph`).
- Не лезет в КД, не пишет в Google Sheets, не вызывает Синтека-API
  без явного запроса пользователя.

**Делает сам:**

- Понимает запрос пользователя (тип задачи: новый расчёт / частичный
  пересчёт / сверка / статус / выгрузка артефактов).
- Знает **карту Drive** (раздел 4) и **каталог навыков** (раздел 5).
- Знает **состояние проекта в репо** (`objects/<имя>/`) и **историю
  бэкапов** (раздел 6) — может прочитать последний бэкап чтобы войти в
  контекст.
- Решает какого сабагента вызвать, в каком порядке, с какими параметрами.
- Сводит результаты в единую таблицу/отчёт.
- Проверяет **тождество**: единая расценка + ВСП ≈ факт (если факт есть).
- Ведёт **журнал расчёта** в `objects/<имя>/journal.md` — что когда
  запускалось, что вышло, что осталось.
- Если нужного сабагента/навыка **нет** — честно говорит «нет, нужен
  навык X, давайте создадим». Не выдумывает.

### 3.2 Типовые сценарии (агент обязан их распознавать)

1. **«Посчитай <объект> АППЗ»** — полный pipeline по АППЗ:
   shahmatka → kvartirograph → spec (АППЗ) → cable + equip + vsp-pnr → сводка.
2. **«Посчитай <объект> все системы»** — все 7 систем: на стороне
   `agent-spec` режим «все системы» + расчётчики по очереди.
3. **«Собери спецификацию <система> из <PDF/папки>»** — только `agent-spec`.
4. **«Какая квартирография в <объект>»** — только `agent-kvartirograph`.
5. **«Где мы остановились по <объект>»** — читает
   `objects/<объект>/journal.md` и/или последний релевантный бэкап в
   `Backup/backupchat/Спецификация/`, отвечает кратко.
6. **«Какие у меня есть навыки/агенты»** — выводит каталог из раздела 5,
   плюс сверяет с актуальным состоянием Drive (`Skills/`) — если на Drive
   есть навык, которого нет в каталоге, говорит «есть `X`, я его не знаю».
7. **«Сверь расчёт с фактом Лофта»** — берёт «Спец Лофт» из файла
   `Расчет этажа двумя методиками ОПБ и КД` как эталон.
8. **«Нужен новый навык X»** — запускает скилл `skillcreator` (если он
   доступен в текущем окружении) либо честно говорит «навык
   `skillcreator` нужен в claude.ai, в Code пока нет — создаём по ТЗ
   вручную».

### 3.3 Что Начальник проверяет перед запуском

- Папка объекта в `objects/<имя>/` существует (если нет — создаёт).
- На Drive есть исходные PDF (папка объекта в `Проекты компании/`).
- Доступны нужные навыки (локально или fallback с Drive).
- Если расчётчик требует базу данных (`База_3_ОПБ` для матчинга
  номенклатуры или для методики ВСП/ПНР) — **наличие xlsx-выгрузки** в
  чате/локально. Если нет — просит пользователя выгрузить (Файл →
  Скачать → Microsoft Excel).

## 7. Технические требования к файлу `agent-boss.md`

### 7.1 Расположение

- Локально: `.claude/agents/agent-boss.md`
- Бэкап на Drive: `Skills/agent-boss/v1.0/agent-boss_v1.0.md`
  (создать обе папки `agent-boss/` и `v1.0/` в `Skills/`).

### 7.2 Формат файла (как у соседних агентов)

```yaml
---
name: agent-boss
description: >-
  Используй ВСЕГДА когда пользователь хочет посчитать ОПБ-расчёт
  по объекту целиком, узнать состояние проекта, выбрать нужного
  сабагента, свести расчёт, сверить тождество с фактом, либо
  спрашивает «где мы», «что у нас по объекту», «какие у меня
  есть навыки», «посчитай <объект>», «собери всё по <объект>».
  Оркестратор семейства сабагентов: знает каталог Drive, каталог
  навыков и обёрток, историю бэкапов, и состояние проектов в
  репо. Не считает сам — дирижирует сборщиками
  (agent-shahmatka / agent-kvartirograph / agent-spec) и
  расчётчиками (agent-cable / agent-equip / agent-vsp-pnr),
  потом сводит результат.
tools: search_files, download_file_content, read_file_content, bash_tool, view, create_file, present_files
model: opus
---

<тело — см. 7.3>
```

**Замечания:**

- В `tools` добавлен `read_file_content` (нужен для чтения Google Sheets
  и больших pdf через MCP) — у других агентов его пока нет, но Боссу нужен.
- `model: opus` — задача оркестратора требует сильной модели.
- Длина `description` ≤ 1024 символов (как и у навыков).

### 7.3 Тело файла (структура)

Тело должно быть **средней длины** — не такой жирный текст, как ТЗ, но и
не три абзаца. Обязательные разделы:

1. **Кто я** — одна фраза + границы (что НЕ делаю).
2. **Карта Drive** — кратко, с ключевыми id (Skills, Backup, Проекты
   компании, ОПБ, плюс главные xlsx, плюс «КД — стоп-зона»). Можно
   ссылаться на отдельный файл `knowledge/drive_map.md`, если он
   создан (создание файла — опционально, см. п. 8).
3. **Каталог обёрток** — таблица «обёртка → навык → назначение → статус».
4. **Pipeline типового объекта** — 3 фазы (сбор → расчёт → сводка).
5. **Сценарии распознавания запроса** — 8 типовых сценариев (см. 3.2).
6. **Алгоритм запуска**:
   - На входе запроса понять сценарий.
   - Проверить наличие нужных обёрток в `.claude/agents/`.
   - Проверить наличие исходных данных на Drive.
   - Запустить нужных сабагентов **через Task tool / делегирование**
     (точный механизм оркестрации см. документацию Claude Code).
   - Собрать результаты, свести.
   - Записать журнал в `objects/<имя>/journal.md`.
   - Отдать артефакты через `present_files`.
7. **Стоп-зоны и грабли** (КД, не геолокация, не выдумывать риски,
   1 уточняющий вопрос максимум).
8. **Что делать если чего-то не хватает** — честный ответ «у меня нет
   X, нужен Y, давайте создадим», не выдумка.

### 7.4 Длина тела

Ориентир — 150–250 строк. Достаточно чтобы покрыть всё выше, но не
дублировать SKILL.md и ТЗ. Главное — **триггеры в `description`** и
**алгоритм оркестрации в теле**.

## 11. Контрольные граблы для исполнителя (не повторять прошлых ошибок)

1. **Не выдумывать новые id.** Все id Drive в разделе 4 — актуальны на
   2026-06-08. Если в репо после старта не нашёл — перепроверить через
   `mcp__Google_Drive__search_files`, а не угадывать.
2. **Не дублировать SKILL.md внутри агента.** Триггеры берутся из
   description навыка; алгоритм — короткий, без копирования содержимого.
3. **Не лить «маркер» вместо полного дампа** при бэкапе. Дамп Босса
   небольшой (≤20 KB) — должен влезть в один `create_file` вызов.
4. **Не использовать `mkdir -p {a,b,c}`** через bash_tool — фигурные
   скобки не разворачиваются. Папки создавать отдельными командами.
5. **На Google Drive `search_files` не поддерживает `'X' in parents`** —
   только `parentId = 'X'`.
6. **Read file content для Google Sheets смешивает листы** — для
   конкретных листов скачивать xlsx и парсить openpyxl.
7. **На Drive нет операции delete через MCP.** Если создал лишний
   файл — оставить, пользователь удалит руками.
